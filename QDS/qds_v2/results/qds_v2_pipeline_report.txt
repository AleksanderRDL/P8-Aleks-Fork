AIS-QDS v2 Pipeline Trace and Logical Soundness Report
======================================================

Project path: QDS/qds_v2
Report date: 2026-04-28


1. Executive verdict
--------------------

AIS-QDS v2 is structurally coherent as an experimental query-driven trajectory
simplification pipeline. The implementation has a clear flow:

  data loading -> typed query generation -> heuristic point-label construction
  -> trajectory-window model training -> query-aware point scoring -> trajectory
  simplification -> query-answer F1 evaluation.

The outer evaluation objective is sensible: simplify trajectories and measure
whether typed query answers are preserved. However, the model is not trained
directly against that same objective. It is trained against heuristic per-point,
per-query-type labels. These labels are often correlated with query relevance,
but they are not a true leave-one-out estimate of how much each point affects
query-answer F1.

Conclusion: the project is logically sound as a prototype/scaffold, but not yet
mathematically rigorous as a query-preservation learner. The weakest link is the
label objective, not the pipeline wiring.

3. High-level experiment flow
-----------------------------

The command line entrypoint parses arguments, builds an ExperimentConfig,
loads data, and calls run_experiment_pipeline(...).

The pipeline performs these steps:

  1. Derive deterministic sub-seeds from the base seed.
  2. Split trajectories into train/test, or use separate train/eval CSV files.
  3. Flatten trajectories into a point tensor and keep trajectory boundaries.
  4. Generate a train typed-query workload from train trajectories.
  5. Generate an eval typed-query workload from eval/test trajectories.
  6. Compute heuristic typed importance labels on train points.
  7. Fit min-max scalers for point and query features.
  8. Train a query-conditioned transformer on trajectory-local windows.
  9. Score eval/test points with the trained model.
  10. Simplify each trajectory with per-trajectory top-k selection.
  11. Execute all eval queries on full and simplified data.
  12. Report per-type and aggregate query-answer F1.

Let the dataset contain T trajectories. Trajectory i has n_i points:

  X_i = (x_i1, x_i2, ..., x_i,n_i)

Flattened points:

  P = (p_1, p_2, ..., p_N)
  N = sum_i n_i

Boundary list:

  B_i = [start_i, end_i)

The boundary list is important because training, inference, simplification, and
evaluation are trajectory-local. The model should not attend across vessel
trajectories.


4. AIS point representation
---------------------------

Each loaded AIS point has 8 columns:

  x_n = [time, latitude, longitude, speed, heading, is_start, is_end, turn_score]

where:

  time       = seconds from the first timestamp in the loaded file
  latitude   = decimal degrees
  longitude  = decimal degrees
  speed      = SOG/speed after cleaning invalid values
  heading    = heading/COG in [0, 360)
  is_start   = 1 for the first point of a trajectory, else 0
  is_end     = 1 for the final point of a trajectory, else 0
  turn_score = normalized absolute heading change

Turn score is computed as:

  delta_heading_j = min(|heading_j - heading_(j-1)|,
                        360 - |heading_j - heading_(j-1)|)

  turn_score_j = delta_heading_j / 180

The baseline model uses the first 7 columns. The turn-aware model uses all 8.
The turn-aware class is currently a thin wrapper around the same architecture.


5. Typed query workload generation
----------------------------------

The workload is a list of typed queries:

  W = {(q_m, c_m)} for m = 1..M

where c_m is one of:

  range, knn, similarity, clustering

The workload mix is normalized:

  pi_c = raw_weight_c / sum_d raw_weight_d

Query counts are allocated approximately as:

  M_c = floor(pi_c * M)

with remaining queries assigned to under-filled types.

If target_coverage is supplied, queries are generated one at a time until the
union of query-covered points reaches the requested fraction or max_queries is
hit.

Each heterogeneous query is padded into a 12-dimensional numeric vector. A
separate type ID is also stored so the model can embed the query type.


6. Query execution mathematics
------------------------------

6.1 Range query

A range query has spatial and temporal bounds:

  q = [lat_min, lat_max, lon_min, lon_max, t_start, t_end]

Point n is inside q if:

  I_range(n, q) = 1[
      lat_min <= lat_n <= lat_max
      and lon_min <= lon_n <= lon_max
      and t_start <= t_n <= t_end
  ]

The range query answer is the set of trajectory IDs that have at least one
point inside the box:

  R_range(q, P) = {i : exists n in trajectory i such that I_range(n, q) = 1}


6.2 kNN query

A kNN query has:

  q = [lat_q, lon_q, t_center, t_half_window, k]

Candidates are points inside the time window:

  C(q) = {n : t_center - t_half_window <= t_n <= t_center + t_half_window}

For each candidate point:

  d_space(n, q) = haversine_km((lat_n, lon_n), (lat_q, lon_q))

  d_time(n, q) = |t_n - t_center|

  d(n, q) = d_space(n, q) + 0.001 * d_time(n, q)

The query selects the k points with smallest d(n, q), then returns trajectory
IDs containing those points:

  R_knn(q, P) = {trajectory_id(n) : n is among top-k nearest candidates}


6.3 Similarity query

A similarity query contains a centroid, time range, radius, top_k, and a short
reference snippet:

  q = [lat_centroid, lon_centroid, t_start, t_end, radius, top_k, reference]

For each trajectory, the implementation takes the segment inside the time
window. It rejects trajectories whose segment centroid is farther than radius
from the query centroid. Remaining trajectories are ranked by a lightweight
DTW-like distance to the reference.

The intended distance is approximately:

  Resample segment A and reference B to a common length L.

  d_sim(A, B) = (1 / L) * sum_l ||A_l - B_l||_2

The returned result is the top_k trajectory IDs with smallest d_sim.

Important implementation note: the current code forms segment = traj[:, :3]
and then computes norm(pa[:, :2] - pb[:, :2]). Since columns [:3] are
[time, lat, lon], pa[:, :2] means [time, lat], not [lat, lon]. If the intended
similarity is spatial trajectory shape, this is likely a bug.


6.4 Clustering query

A clustering query has a range-like box plus DBSCAN parameters:

  q = [lat_min, lat_max, lon_min, lon_max, t_start, t_end, eps, min_samples]

For each trajectory i with points inside the query box, compute a representative
lat/lon centroid:

  r_i(q) = mean_{n in trajectory i and inside q} [lat_n, lon_n]

Then run a simple DBSCAN over the representative points r_i(q). The result is a
label per trajectory:

  R_cluster(q, P) = [label_1, label_2, ..., label_T]

Noise label -1 is ignored during clustering F1 pair construction.


7. Training label construction
------------------------------

The model trains on a label matrix:

  Y in R^(N x 4)

where Y[n, c] is the heuristic importance of point n for query type c.

There is also a labelled mask:

  M_label in {0, 1}^(N x 4)

The labels are generated from the train workload, not the eval workload.


7.1 Range labels

For a range query q, let:

  S(q) = {n : I_range(n, q) = 1}

  base(q) = sum_{r in S(q)} speed_r

For each point inside the query:

  Y[n, range] += |speed_n / max(|base(q)|, eps)|

  M_label[n, range] = 1

Interpretation: this labels speed contribution inside range boxes. It does not
directly label whether point n is necessary to preserve a trajectory-ID query
answer.


7.2 kNN labels

For a kNN query, compute d(n, q) for all time-window candidates. Let d_k be the
distance of the k-th nearest candidate:

  scale = d_k + eps

Candidate weight:

  w(n, q) = 1 / (1 + d(n, q) / scale)

Then:

  Y[n, knn] += w(n, q)

Points within 3 * scale are marked labelled.

This is a reasonable soft proxy for kNN relevance.


7.3 Similarity labels

For a similarity query, candidate points are inside the query time/radius
region. A random sample of candidates is measured. Let the reference spatial
centroid be:

  mu_ref = mean_r [lat_r, lon_r]

For measured candidate point n:

  d_ref(n) = ||[lat_n, lon_n] - mu_ref||_2

  Y[n, similarity] += 1 / (1 + d_ref(n))

This is a centroid-proximity heuristic. It does not directly approximate the
effect of removing point n on the top-k trajectory similarity answer.


7.4 Clustering labels

For clustering queries, candidate points are sampled inside the query box. The
label is speed normalized by global max speed:

  Y[n, clustering] += speed_n / (max_r speed_r + eps)

This is the least logically aligned label. The actual clustering query clusters
trajectory representatives in latitude/longitude space, while the label is a
speed feature. Unless speed is intentionally used as a behavioral proxy, this
does not match the clustering objective.


7.5 Interpolation and speed blending

After sparse labels are added, unlabelled entries are filled within each
trajectory using the nearest measured label for that query type.

Then each type receives a global speed baseline:

  speed_norm(n) = (speed_n - min_r speed_r) / (max_r speed_r - min_r speed_r + eps)

  Y[n, c] <- 0.90 * Y[n, c] + 0.10 * speed_norm(n)

The labelled mask is then set to true for every point and every type:

  M_label[n, c] = 1 for all n, c

Finally, each type column is min-max normalized:

  Y[n, c] <- (Y[n, c] - min_r Y[r, c]) / (max_r Y[r, c] - min_r Y[r, c] + eps)

This gives dense gradients and stabilizes training, but it also means all heads
learn at least some speed signal, even when the workload does not strongly
support a query type.


8. Feature scaling
------------------

The FeatureScaler fits min and max values on train points and train query
features:

  p_min[f] = min_n P[n, f]
  p_max[f] = max_n P[n, f]
  q_min[g] = min_m Q[m, g]
  q_max[g] = max_m Q[m, g]

At training and inference:

  P_norm[n, f] = (P[n, f] - p_min[f]) / max(p_max[f] - p_min[f], eps)

  Q_norm[m, g] = (Q[m, g] - q_min[g]) / max(q_max[g] - q_min[g], eps)

The scaler is saved with checkpoints, which is correct and necessary for
consistent inference.


9. Model architecture and forward pass
--------------------------------------

Let point feature dimension be Fp and query feature dimension be Fq. The model
embedding dimension is D.

Point encoder:

  E_p[n] = Linear2(Dropout(ReLU(Linear1(P_norm[n]))))

Add sinusoidal positional encoding:

  H0[n] = E_p[n] + PE[n]

Trajectory-local transformer encoder:

  H = TransformerEncoder(H0, padding_mask)

Query type embedding:

  e_type[m] = Embedding(type_id[m])

Query encoder:

  Z[m] = QueryEncoder(concat(Q_norm[m], e_type[m]))

Cross-attention from points to queries:

  C = MultiHeadAttention(query=H, key=Z, value=Z)

The implementation chunks queries and accumulates attention outputs, then
divides by the total query count M:

  C_impl = (1 / M) * sum_chunks MHA(H, Z_chunk, Z_chunk)

Important note: standard attention already normalizes over keys inside each
MHA call. If all queries fit in one chunk, dividing by M likely shrinks query
context too much. This may weaken query conditioning.

Final per-type heads:

  y_hat[n, c] = head_c(H[n] + C[n])

The output is one score per point per query type:

  Y_hat in R^(N x 4)

It is not one score per point per individual query.


10. Windowing and no trajectory leakage
---------------------------------------

Training and inference use fixed-length trajectory-local windows. For a
trajectory longer than window_length, overlapping windows are created using
window_stride. For shorter trajectories, zero padding is added and padding_mask
marks invalid positions.

This avoids cross-trajectory attention. During inference, overlapping window
predictions are averaged:

  pred[n, c] = sum_w pred_w[n, c] / count_w(n)

This is a good design choice.


11. Training objective
----------------------

Training uses pairwise ranking plus a small MSE anchor.

For each query type c, valid points are sampled. Let a and b be a sampled pair.
The desired ordering sign is:

  s_ab = sign(Y[a, c] - Y[b, c])

The margin ranking loss is:

  L_rank(a, b, c) = max(0, margin - s_ab * (Y_hat[a, c] - Y_hat[b, c]))

MSE anchor:

  L_mse(c) = mean_n (Y_hat[n, c] - Y[n, c])^2

Total batch loss is approximately:

  L = sum_c (L_rank(c) + 0.05 * L_mse(c))
      + lambda * mean(Y_hat^2)

where lambda is l2_score_weight.

The model_config.epochs value is clamped:

  effective_epochs = max(8, configured_epochs)

Important note: an epoch workload mix is sampled, but in the current code it is
only checked for being greater than zero. It is not multiplied into the loss.
Since the sampled weights are always positive, all four heads effectively train
each epoch. The comments imply type weighting, but the implementation does not
currently apply it as a weight.


12. Inference and simplification
--------------------------------

For eval/test data, the trained model predicts:

  Y_hat[n, c]

The workload mix is normalized into pi_c. A scalar point score is computed:

  score[n] = sum_c pi_c * Y_hat[n, c]

If query buffers are enabled, a point is marked near a query when its lat/lon is
inside a buffered query geometry. This buffer currently ignores the query time
window.

If query_area_boost != 1:

  score[n] <- score[n] * query_area_boost, for near-query points

Then query-first selection is used. For each trajectory i:

  k_i = ceil(compression_ratio * n_i)

  k_i = max(2, min(k_i, n_i))

If the trajectory has no near-query points, keep top-k_i by score. If it has
near-query points and their count is <= k_i, keep those near-query points only,
plus pinned endpoints. If near-query count > k_i, keep the top-k_i near-query
points by score.

Without query-first mode, simplification uses deterministic top-k within each
trajectory:

  retained_i = top_k(score[start_i:end_i], k_i)

First and last points are always retained.

Because query-first selection may keep fewer than k_i non-endpoint points, the
actual retained compression can be below the requested ratio.


13. Evaluation metrics
----------------------

Evaluation compares original query answers to simplified query answers.

For range, kNN, and similarity queries, answers are treated as sets of
trajectory IDs:

  R_o = query(original_points)
  R_s = query(simplified_points)

Precision:

  precision = |R_s intersection R_o| / |R_s|

Recall:

  recall = |R_s intersection R_o| / |R_o|

F1:

  F1 = 2 * precision * recall / (precision + recall)

Special cases:

  if R_o and R_s are both empty: F1 = 1
  if exactly one is empty:       F1 = 0

For clustering, labels are converted into same-cluster co-membership pairs,
excluding noise label -1. F1 is computed over those pair sets.

Per-type F1:

  F1_c = mean_{q in workload, type(q)=c} F1(q)

Aggregate F1:

  F1_aggregate = sum_c pi_c * F1_c

Current source code reports higher-is-better F1. Some saved artifacts under
QDS/qds_v2/results still use old AggregateErr/aggregate_error fields and should
be considered stale relative to the current code.


14. Logical strengths
---------------------

1. The pipeline is end-to-end complete.

   Data, query generation, training, inference, simplification, and evaluation
   are connected in a coherent order.

2. Evaluation uses the right high-level target.

   It compares query answers before and after simplification, which is exactly
   the behavior a query-driven simplifier should care about.

3. Trajectory boundaries are respected.

   Windowing avoids attention across different vessels, and simplification is
   per trajectory.

4. The scaler is persisted.

   This avoids a common training/inference mismatch.

5. The top-k simplifier has deterministic tie handling.

   Equal scores should not systematically bias toward early positions.

6. The code has regression tests for important contracts.

   Tests cover metric polarity, query type IDs, scaler persistence, no
   cross-trajectory leakage, top-k tie behavior, query coverage generation, and
   training collapse.


15. Main logical weaknesses
---------------------------

1. Training labels do not directly optimize query-answer preservation.

   The evaluation metric is query-answer F1. The training target is a heuristic
   point score. This is the central conceptual gap.

2. Range labels are speed-mass labels, but range evaluation is trajectory-ID
   membership.

   A slow point can be essential if it is the only retained point proving a
   trajectory entered a query box. A fast point can be redundant if many nearby
   retained points are also inside the same box.

3. Clustering labels are speed-based, while clustering execution is spatial.

   The query clusters lat/lon representatives. The label uses speed. This is
   hard to justify mathematically unless speed is an intentional domain proxy
   for cluster importance.

4. Similarity distance appears to use time+latitude rather than latitude+longitude.

   The code slices [time, lat, lon] and then uses the first two columns for the
   norm. If the desired similarity is geographic shape, this should likely use
   columns [lat, lon].

5. The model predicts per query type, not per query.

   The output Y_hat[n, c] cannot directly express that point n is important for
   one range box but irrelevant to another range box. Query conditioning enters
   through aggregate query attention and query-buffer postprocessing.

6. Query context may be over-shrunk.

   The cross-attention output is divided by the number of queries M. With one
   chunk, MHA already returns a normalized context, so the extra division likely
   weakens query conditioning.

7. The epoch workload mix is not actually used as a numeric loss weight.

   The implementation samples epoch_mix but only checks whether each component
   is positive. Since the sampled components are positive, all heads train
   every epoch without those weights affecting the loss scale.

8. The oracle evaluation is not directly comparable to test methods.

   The diagnostic oracle uses train labels and train points/workload, while
   MLQDS and baselines are evaluated on test/eval points. Treat it as a sanity
   diagnostic, not a fair upper bound.

9. Empty query results can inflate scores.

   If both original and simplified query answers are empty, F1 is 1. This is
   mathematically conventional for set agreement, but it can make workloads
   with many empty queries look better than they are.

10. Query-first buffer ignores time.

   Spatial query buffers can prioritize points near a query geometry even if
   their timestamp is outside the query time window.


16. How the model fits to the data
----------------------------------

The model fits by learning a function:

  f_theta(P_norm, Q_norm, type_ids) -> Y_hat

where:

  P_norm is the normalized point sequence for a trajectory window
  Q_norm is the normalized query workload
  type_ids identify query types
  Y_hat[n, c] is a learned importance score for point n and query type c

The parameters theta include:

  point encoder weights
  transformer encoder weights
  query type embedding weights
  query encoder weights
  cross-attention weights
  four per-type output heads

The optimizer is Adam with learning rate model_config.lr. Gradients come from
ranking sampled point pairs and a small MSE anchor to the heuristic labels.

Conceptually, the model learns to rank points so that:

  if Y[a, c] > Y[b, c], then Y_hat[a, c] should be at least margin larger than
  Y_hat[b, c]

The final simplifier then assumes that the highest predicted scores are the
points most worth retaining under the eval workload mix.


17. Overall assessment
----------------------

The project is useful and internally organized. It has the right components for
query-driven trajectory simplification. The current implementation can support
experiments and comparisons.

The strongest mathematical criticism is that the training target is not the
evaluation target. A more rigorous version would define each point label by the
loss in query-answer quality caused by dropping that point, or by dropping a
local neighborhood/window around that point. For example:

  importance(n, c) = E_{q of type c} [F1(query(P), query(P without n)) loss]

or, more explicitly:

  importance(n, c) = E_q [1 - F1(R(q, P), R(q, P \ {n}))]

This would align labels with the actual query-preservation metric. It would be
more expensive, but it would make the learned objective much more defensible.

Until that alignment is improved, AIS-QDS v2 should be described as a
heuristic query-conditioned learned simplifier, not a fully principled optimizer
of query-answer preservation.


18. Recommended fixes or next steps
-----------------------------------

1. Fix or verify similarity distance dimensions.

   If geographic similarity is intended, compute distance on [lat, lon], not
   [time, lat].

2. Replace clustering speed labels with spatial cluster-sensitivity labels.

   A better label would measure whether retaining/removing a point changes the
   representative centroid enough to alter co-membership pairs.

3. Build true query-impact labels for at least range and kNN.

   For each point or short window, estimate:

     delta(n, q) = 1 - F1(R(q, P), R(q, P without point/window n))

   Then aggregate over queries by type.

4. Make the model score point-query pairs, or at least query-specific pooled
   type scores.

   Current output is per type. Per-query scoring would better match query-
   driven simplification.

5. Revisit cross-attention normalization.

   Remove the division by total query count, or use a mathematically correct
   chunked attention normalization if chunking is needed.

6. Apply epoch_mix as an actual loss multiplier, or remove it.

   Current comments and implementation do not match.

7. Make query-first buffer spatiotemporal.

   Near-query should include both spatial geometry and time constraints.

8. Refresh result artifacts.

   Existing saved result files with AggregateErr are stale relative to current
   source code, which reports AggregateF1.

9. Treat the oracle as train-diagnostic only, or implement a fair test oracle.

   A fair oracle should compute labels on the same evaluation points and
   workload used by the other methods.

10. Report query non-empty rates.

   Because empty-vs-empty F1 is 1, result tables should include the fraction of
   queries with non-empty original answers.


19. Final short conclusion
--------------------------

AIS-QDS v2 has a coherent experimental pipeline and sensible evaluation shell,
but its current mathematical core is proxy-based. The model fits heuristic
importance labels, then uses those learned labels to rank points for retention.
That can produce useful behavior, especially for range/kNN workloads, but the
labels need to be aligned more directly with query-answer F1 before the project
can be described as logically rigorous.
